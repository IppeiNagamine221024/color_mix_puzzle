> **対象バージョン**: v1.1（初回公開 v1.0.0 完了後）

> **関連ドキュメント**: [要件定義（本体）](../requirements.md)

## 1. 概要

### 1.1 背景

App Store 初回公開が完了した。次回アップデートでは、拡散導線の追加と、新規ユーザーの理解しやすさ向上を目的とした UX 改善を行う。

### 1.2 スコープ（今回のアップデート）

| # | 機能 | 概要 |
|---|------|------|
| 1 | **シェア機能** | クリア時に、OS 共有シート経由で SNS 等へ共有できる導線を追加する |
| 2 | **遊び方ページの充実** | テキスト中心の遊び方画面を画像ベースに差し替え、初回起動時に同等内容のポップアップを表示する |
| 3 | **軽微なバグ修正・UX 改善** | exit スキップ後の enter 再生ずれ修正、広告回復の確認ダイアログ、行動ボタン選択状態の強調 |

### 1.3 スコープ外（今回やらないこと）

- 新ステージ追加・ゲームルール変更
- 新しい課金商品の追加
- Web 版・多言語対応
- シェア先 SNS の追加（TikTok 等）は将来拡張とする
- **アプリ内レビュー依頼**（StoreKit / Play の評価ダイアログ）は将来計画とする（§10）

## 2. 機能要件 — シェア機能

### 2.1 目的

ユーザーがクリア達成などの達成感を外部に共有しやすくし、自然な口コミ・再訪問を促す。

### 2.2 想定シェア先（初版）

| サービス | 優先度 | 備考 |
|---------|--------|------|
| **X（旧 Twitter）** | 高 | テキスト + スクリーンショット画像 |
| **Instagram** | 高 | スクリーンショット画像（フィード / ストーリーズ） |
| **LINE** | 高 | テキスト + スクリーンショット画像 |

### 2.3 シェアのトリガー（表示タイミング）— **確定**

| タイミング | 説明 |
|-----------|------|
| **ステージクリア後** | **クリア Lottie 演出の後**。「タップで続ける」で Lottie を閉じた直後、退場アニメーションの前にシェア UI を表示する |

初版の最小構成は **クリア時の 1 箇所** のみとする。

#### 操作フロー（確定）

```
クリア演出（Lottie）
  → アニメ終了後「タップで続ける」表示
  → 右上に「SNSでシェア」ボタン（任意）
      ├─ タップ → プレビュー +「シェアする」
      │            └─ OS 共有シートで共有先を選択 / 「閉じる」でクリア画面に戻る
      └─ 画面タップ（続ける）→ 退場アニメ → メイン画面
```

シェアは強制せず、クリア後の任意導線とする。
### 2.4 シェア内容

#### 2.4.1 共有テキスト（確定案）

```
Color Order でステージ {N} をクリアしました！
#ColorOrder
https://apps.apple.com/us/app/color-order/id6788045840
```

- ステージ番号 `{N}` は動的に差し込む
- ハッシュタグ・ストア URL は定数化する
- Android 公開後はプラットフォームに応じて Play Store URL も記載する（初版は iOS のみ）

#### 2.4.2 共有画像 — **確定**

| 項目 | 内容 |
|------|------|
| 方式 | **動的生成（クリア時点のゲーム画面スクリーンショット）** |
| キャプチャ対象 | クリア Lottie を閉じた直後のゲーム画面（盤面・課題パターン・ステージ情報が見える状態） |
| キャプチャタイミング | Lottie オーバーレイ非表示後、シェア UI 表示前 |
| 形式 | PNG |
| 一時保存 | `expo-file-system` のキャッシュディレクトリ等に書き出し、共有後に削除してよい |

> 汎用バナー画像は採用しない。ユーザー固有のクリア画面が共有されることが訴求力の中心となる。

#### 2.4.3 キャプチャ時の注意

- Lottie オーバーレイ・シェア UI 自体はキャプチャに含めない
- クリア演出中の `patternGlow`（課題一致のグロー）が有効な状態をキャプチャする
- キャプチャ失敗時はテキストのみ共有にフォールバックし、クラッシュしない

### 2.5 UI 要件

#### クリア後シェア UI

- クリア Lottie 完了後の画面（「タップで続ける」）右上に **「SNSでシェア」** ボタンを表示する
- ボタンタップでプレビューと **「シェアする」** を表示する
- 「シェアする」で OS 共有シートを開き、ユーザーが共有先（X / Instagram / LINE 等）を選ぶ
- シェアシートは **「閉じる」** で閉じ、クリア画面に戻る（退場はしない）
- シェアせずに画面タップで従来どおり退場 → メイン画面へ進める
- 既存の wood テーマ（`constants/Theme.ts` / `constants/wood.ts`）に合わせた見た目とする

### 2.6 技術方針 — **確定**

| レイヤー | ライブラリ | 状態 |
|---------|-----------|------|
| スクリーンショット | **`react-native-view-shot`** | 採用・実装済み |
| SNS 共有 | **`react-native-share` の `Share.open`（OS 共有シート）** | 採用・実装済み |

SNS 別ボタン（X / Instagram / LINE）は採用しない。共有先の選択は OS に委譲する。

### 2.7 受け入れ条件

- [ ] クリア Lottie の「タップで続ける」後に「SNSでシェア」が表示される
- [ ] 「シェアする」で OS 共有シートが開く
- [ ] 共有画像はクリア時点のゲーム画面スクリーンショットである
- [ ] シェアを閉じたあと、画面タップで従来どおりメイン画面へ戻れる
- [ ] 共有テキストにステージ番号と App Store URL が含まれる
- [ ] シェア失敗時（キャンセル等）にクラッシュしない
- [ ] シェアの有無にかかわらず、クリア記録・スタミナ・進行データに影響しない

## 3. 機能要件 — 遊び方ページの充実

### 3.1 目的

現状のテキストのみの遊び方画面はとっつきづらい。視覚的な説明画像に差し替え、初回ユーザーがルールを素早く理解できるようにする。

### 3.2 現状

| 項目 | 内容 |
|------|------|
| 画面 | `app/settings/how-to-play.tsx` |
| 導線 | 設定 → 「遊び方」 |
| 構成 | `STEPS` 配列によるテキストカード 5 枚（目的 / スライド / 混合 / 入替 / スタミナ） |
| 初回ポップアップ | **なし** |

### 3.3 変更後の遊び方画面

#### 3.3.1 コンテンツ — **確定（アセット配置済み）**

テキストカードを **説明画像** に差し替える。画像は `assets/images/how-to-play/` に配置済み。

| 順序 | ファイル | 内容 |
|------|---------|------|
| 1 | `htp_purpose.png` | ゲームの目的説明 |
| 2 | `htp_slide.png` | スライド操作の説明 |
| 3 | `htp_mix.png` | 混合操作の説明 |
| 4 | `htp_replace.png` | 入替操作の説明 |
| 5 | `hint.png` | ヒント機能の説明 |

> 旧テキスト版にあった「スタミナ」説明は画像セットに含まれない。スタミナはチュートリアル（ステージ 1〜5）で自然に学ぶ前提とする。

#### 3.3.2 レイアウト

| 要素 | 説明 |
|------|------|
| ヘッダー | 既存の `SettingsSubHeader`（タイトル「遊び方」）を継続 |
| 本文 | 縦スクロール。上表の順に画像を配置 |
| 画像表示 | `Image` + `resizeMode="contain"`（画面幅に合わせる） |
| アクセシビリティ | 各画像に `accessibilityLabel` を設定（上表の内容を要約） |

#### 3.3.3 アセット一覧

```
assets/images/how-to-play/
  htp_purpose.png
  htp_slide.png
  htp_mix.png
  htp_replace.png
  hint.png
```

遊び方画面と初回ポップアップで **同一セット・同一順序** を使う。

### 3.4 初回起動ポップアップ

#### 3.4.1 目的

初回インストール直後のユーザーに、設定画面を開かなくてもルールを伝える。

#### 3.4.2 表示タイミング — **確定**

| 条件 | 挙動 |
|------|------|
| **新規インストール後の初回起動** | メイン画面のデータ読み込み完了後、**1 回だけ** ポップアップを表示 |
| **既存ユーザー（v1.0 → v1.1）** | **表示しない**（セーブデータが存在する場合） |
| **2 回目以降の起動** | 表示しない |

#### 3.4.3 ポップアップの内容

- **遊び方画面と同等の内容**（`htp_*.png` + `hint.png` の 5 枚）
- フルスクリーンまたはモーダル形式のオーバーレイ
- 縦スクロールで全画像を閲覧可能
- **「はじめる」ボタン**（画面下部固定）で閉じる。背景タップでは閉じない（誤タップ防止）

#### 3.4.4 表示場所

- メイン画面の上にオーバーレイとして表示
- ポップアップ表示中はステージ選択・設定などの操作をブロック
- 閉じた後は通常のメイン画面操作に戻る

#### 3.4.5 再表示

- 設定から再表示する機能は今回スコープ外
- ルールの再確認は「設定 → 遊び方」画面を使う

### 3.5 永続化（初回表示フラグ）— **確定**

| 項目 | 内容 |
|------|------|
| キー | `has_seen_how_to_play_intro` |
| 保存先 | 既存の MMKV / AsyncStorage 抽象（`src/storage/`） |
| 表示条件 | `has_seen_how_to_play_intro` が未設定 **かつ** セーブデータが無い（真の新規） |
| 更新タイミング | ポップアップの「はじめる」タップ時に `true` にする |

**既存ユーザー（v1.0 → v1.1）**: セーブデータが存在するユーザーには **ポップアップを表示しない**（確定）。

### 3.6 コンポーネント設計

```
components/how-to-play/
  howToPlayImages.ts      # 画像リストと accessibilityLabel 定数
  HowToPlayContent.tsx    # 画像スクロール本体（共通）
  HowToPlayIntroModal.tsx # 初回ポップアップ
```

- `app/settings/how-to-play.tsx` は `HowToPlayContent` をラップする薄い画面にする
- `app/index.tsx` から `HowToPlayIntroModal` を呼び出す

### 3.7 受け入れ条件

- [ ] 設定 → 遊び方で 5 枚の説明画像が順に表示される
- [ ] 画像は縦スクロールで全枚閲覧できる
- [ ] 新規インストール後、初回起動時にポップアップが 1 回表示される
- [ ] ポップアップの内容は遊び方画面と同等である
- [ ] v1.0 からアップデートした既存ユーザーにはポップアップが出ない
- [ ] 遊び方画面は従来どおり設定からいつでも開ける

## 4. 非機能要件

### 4.1 パフォーマンス

- 遊び方画像は `require()` でバンドル同梱とし、起動時の読み込み遅延を最小化する
- シェア用スクリーンショットはクリア時に都度生成し、キャッシュディレクトリに一時保存する

### 4.2 アクセシビリティ

- シェアボタン・ポップアップの「はじめる」ボタンに `accessibilityLabel` を付与する
- 遊び方画像には各画像の要約を `accessibilityLabel` に設定する

### 4.3 分析・ログ

- 初版ではシェア完了の analytics 送信は **任意**（将来用にイベント名を定数化）
  - 例: `share_tap_x`, `share_tap_instagram`, `share_tap_line`, `share_skip`

### 4.4 ストア申請

- シェア機能追加は App Store の「新機能」説明の更新を検討する
- プライバシーポリシーへの追記は不要（ユーザー自身が SNS へ共有するのみ）

## 9. 軽微なバグ修正・UX 改善

### 9.1 exit スキップ後の enter 再生ずれ（バグ）

| 項目 | 内容 |
|------|------|
| 現象 | 退場演出を途中スキップすると、次回入場演出が途中フレームから再生される |
| 原因想定 | enter / exit が同一アセットを共有しており、スキップ時に再生位置がリセットされない |
| 修正 | スキップ・完了時に `reset()` してから解除。入場もフレーム 0 から明示再生する |

### 9.2 広告スタミナ回復の確認ダイアログ（機能追加）

| 項目 | 内容 |
|------|------|
| トリガー | メイン画面の「広告でスタミナ回復」ボタンタップ時 |
| ダイアログ | 「広告を見てスタミナを回復しますか？」 |
| 操作 | 「見る」→ リワード広告表示 / 「キャンセル」→ 何もしない |

### 9.3 混合・入替ボタンの選択状態強調（改善）

| 項目 | 内容 |
|------|------|
| 現状 | 選択時は白枠 → 黒枠のみで判別しづらい |
| 改善 | 選択中は枠の強調・やや拡大・背景を明るくして、選択中であることが一目で分かるようにする |

## 10. 将来計画 — アプリ内レビュー依頼（v1.1 対象外）

> **状態**: 未実施。検証（実機ビルド・本番相当での表示確認）に時間がかかるため、v1.1 では実装しない。将来アップデートで検討する。

### 10.1 目的

満足度の高いタイミングで App Store / Play の標準レビューダイアログを出し、自然な評価獲得を促す。カスタム「星をつけて」UI は使わず、OS 標準 API に任せる。

### 10.2 表示条件（案・確定候補）

次を **すべて満たしたとき** に、レビュー依頼候補とする。

| 条件 | 内容 |
|------|------|
| **ステージ 30 クリア** | **初回クリア時のみ**（`clearedStages` に未含有だったクリア）。再プレイでは対象外 |
| **3 連続ステージクリア** | 直近のクリアが途切れず 3 回続いていること。ゲームオーバーで連続カウントをリセット |
| **未依頼 / 新バージョン** | 当該アプリバージョンでまだ依頼していない。バージョンが上がったら再度候補になりうる |

OS 側にも年間上限（iOS は概ね 365 日あたり最大 3 回など）があり、API を呼んでもダイアログが出ない場合がある（正常）。

### 10.3 表示タイミング（案）

| 項目 | 内容 |
|------|------|
| **出す場所** | クリア後に **ホーム（メイン画面）へ戻ったとき** |
| **重ねない** | クリア Lottie・退場アニメ・シェア UI（`ClearShareOverlay`）と同時表示しない。シェア閉じ済み・退場完了後に依頼する |

### 10.4 技術方針（案）

| 項目 | 内容 |
|------|------|
| パッケージ | `expo-store-review`（`StoreReview.requestReview()`） |
| 永続化 | 連続クリア数、依頼済みフラグ（または最後に依頼した `appVersion`）をセーブ／ローカルに保持 |
| 実装の呼び方 | ボタン直結はしない（Apple / Expo ガイドライン）。条件成立フラグを立て、ホーム `useFocusEffect` 等で 1 回呼ぶ |
| 検証留意 | TestFlight ではダイアログが出ない。開発ビルドと本番で挙動が異なる。ネイティブ再ビルドが必要 |

### 10.5 チェックリスト（実装時）

- [ ] `expo-store-review` 導入・ネイティブ再ビルド
- [ ] 連続クリア / ゲームオーバー時リセット
- [ ] ステージ 30 初回クリア × 3 連続 × バージョン未依頼の判定
- [ ] ホーム復帰時に依頼（シェア・演出と非重複）
- [ ] iOS 実機（可能なら本番相当）での表示確認
- [ ] （任意）設定画面等に App Store の `?action=write-review` リンクを常設
