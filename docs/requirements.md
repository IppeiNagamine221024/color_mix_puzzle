# いろあわせパズル — 要件定義

## 1. 概要

**ゲーム名**: いろあわせパズル（仮。リリース前に変更する可能性あり）  
**英語表記（参考）**: Iroawase Puzzle

三原色（赤・黄・青）をはじめとする色ブロックを盤面上で操作し、混合して、ステージごとの課題パターン（配色の配置）を盤面のどこかに再現したらクリアするパズルゲーム。

1ターンにつき1つの行動を選択する。同じ行動を2回連続で選ぶことはできない。

初期リリースでは約100ステージを用意する。

**プラットフォーム**: スマートフォン向けアプリ（iOS 先行リリース、Android を後続対応）。  
**収益化**: 広告（バナー・リワード）。  
**挑戦権**: スタミナ制（上限3、時間回復 + リワード広告回復）。

---

## 2. 用語


| 用語          | 説明                                        |
| ----------- | ----------------------------------------- |
| ターン         | プレイヤーが行動（A / B / C のいずれか）を1回実行すること        |
| ブロック        | 盤面上の色付きマス。原色・二次色・三次色など、名前の付いた色を持つ         |
| お邪魔ブロック     | 移動・混合・スワップの対象にならない固定マス                    |
| 課題（パターン）    | クリア条件となる色と相対配置のテンプレート                     |
| NEXT        | 行動 B 実行時に出現する次のブロックの色。ゲーム画面に常時表示する        |
| 行動可能ターン     | ステージごとに与えられる行動回数の上限。行動 A / B / C ごとに1減少する |
| スタミナ        | ステージへ挑戦するために消費する挑戦権。上限・回復ルールは第15章参照       |
| チュートリアルステージ | ステージ 1〜5。スタミナを消費しない（第15章参照）               |


---

## 3. 盤面

- 盤面サイズ（幅・高さ）はステージごとに設定可能とする。
- マスには以下のいずれかが存在する。
  - 空き
  - 色ブロック（操作可能）
  - お邪魔ブロック（操作不可）
- 行動 B におけるブロックの移動は、2048 と同様に指定方向へ全ブロックがスライドし、壁・お邪魔ブロック・盤端で停止する。

---

## 4. 行動

各ターンで A / B / C のいずれか1つを選択する。**直前のターンと同じ行動は選択できない。**

### 4.1 行動 A — 混合

- 隣接（上下左右）する2つの**異なる**色ブロックを選択し、1マスに混合する。
- 同色同士の混合は不可。
- 混合は、レシピ一覧に定義された「名前の付いた色」が結果として得られる場合のみ実行可能とする。名前のない色になる組み合わせは **MIX 不可**（操作として拒否する）。
- 混合後、もう一方のマスは空きになる。
- **黒の特殊ルール**
  - 黒と黒を混合した場合のみ、2マスとも消滅する（盤面から除去）。
  - 黒と黒以外の色の混合は不可。

### 4.2 行動 B — スライド＋出現

- 盤面全体を上下左右のいずれか1方向にスライドする。
- スライド後、移動方向の**反対側の辺**上の空きマスに、NEXT の色のブロックが1つ出現する。
- 出現位置は、反対側の辺上の空きマスから**ランダム**に選ぶ。
- スライド実行後、NEXT を次の色に更新する（抽選ルールは第4.5章参照）。
- ゲーム画面上には常に現在の NEXT と残り行動可能ターン数を表示する。

#### ステージ1（チュートリアル）の例外

- ステージ1のみ、出現位置をランダムではなく固定とし、1〜2手でクリアできるように設計する（詳細はステージデータ参照）。
- NEXT の抽選は全ステージ共通でランダムとするが、ステージ開始時の NEXT 初期値はステージデータで指定できる（ステージ1は `red`）。

### 4.3 行動 C — スワップ

- 隣接する2つの**操作可能な色ブロック**の位置を入れ替える。
- 空きマス・お邪魔ブロックとのスワップは不可。

### 4.4 NEXT の更新タイミング

- NEXT の消費・更新は行動 B のときのみ行う。
- 行動 A / C の実行では NEXT は変化しない。

### 4.5 NEXT の抽選ルール（確定）

- 行動 B 実行後の NEXT 色は、**ランダム抽選**で決定する（全ステージ共通）。
- デフォルトの抽選対象は**原色**（赤・黄・青）とし、各原色を均等確率で選ぶ。
- ステージ開始時の NEXT 初期値のみ、ステージデータの `next.initial` で指定できる。
- 抽選対象色の拡張（二次色を含める等）はステージデータまたはゲーム設定で上書き可能とする（初期リリースでは原色のみ）。

### 4.6 行動可能ターン

- 各ステージは `maxTurns`（行動可能ターン数）を持つ。
- 行動 A / B / C を1回実行するたびに、残り行動可能ターンを **1** 減少させる。
- 行動の実行が拒否された場合（混合不可など）は減少しない。

## 5. 色と混合ルール

### 5.1 基本方針

- 混合は一般的な絵の具（減法混色）の考え方に基づく。
- ゲーム内で使用する色は、すべて**名前の付いた色**として定義する（例: 赤、黄、青、橙、緑、紫、朱、黒 など）。
- 原色・二次色・三次色など、レシピで到達可能な色のみ盤面・課題・NEXT に登場しうる。

### 5.2 混合の可否


| 条件              | 結果         |
| --------------- | ---------- |
| レシピ一覧に結果色が存在する  | 混合可能       |
| レシピ一覧に結果色が存在しない | 混合不可       |
| 黒 + 黒           | 両方消滅（行動 A） |
| 黒 + 黒以外         | 混合不可       |
| 同色同士            | 混合不可       |


### 5.3 黒の生成

- 同一マスに赤（R）・黄（Y）・青（B）が**均等に近い比率**で混ざったときのみ、黒が生成される。
- 黒は不要色の安全弁ではなく、主に盤面圧力・ペナルティとして機能する。黒同士の除去は意図的な盤面管理手段とする。

### 5.4 レシピデータ

- 具体的な混合レシピ（例: 赤 + 橙 = 朱）は、ゲームデータとして `data/color_recipes.json`（仮）に一覧定義する。
- 二次色同士・三次色同士の混合も、同じルール（結果が名前付き色として定義されている場合のみ可）に従う。
- レシピ一覧の拡張はデータ追加で行い、ロジック変更を最小化する。

## 6. クリア条件

- ステージごとに定義された**課題パターン**が、盤面上の**どこか**に出現した瞬間に即クリアとする。
- パターン一致時に追加の確認操作（提出ボタンなど）は不要。
- クリア成立後は **光を中心とした派手めの演出**（第6.4章）を再生し、オーバーレイ経由でメイン画面へ遷移する。

### 6.1 パターン一致ルール（デフォルト）


| 項目        | ルール                 |
| --------- | ------------------- |
| 回転        | 不可                  |
| 反転        | 不可                  |
| 配置        | 盤面の任意位置で一致すればよい     |
| 課題外のマス    | パターンに含まれないマスは何色でもよい |
| お邪魔ブロック   | パターンに含めない           |
| 最小パターンサイズ | 2マス                 |


### 6.2 パターンの向き

- 課題の表示どおりの相対配置で一致する必要がある。
- 例: `[青][赤]` は**左が青・右が赤**の横並び。
- 縦並びの `[青]` / `[赤]` は別課題として定義する（横並びとは別物）。

### 6.3 課題パターンのデータ表現

- 課題は「色付きセルの相対座標」のリストとして保持する。
- 例（横2マス `[青][赤]`）:

```json
{
  "cells": [
    { "dx": 0, "dy": 0, "color": "blue" },
    { "dx": 1, "dy": 0, "color": "red" }
  ]
}
```

### 6.4 クリア演出（確定）

クリア演出は **光多め・派手目** を方針とする。ゲームオーバー演出は簡素でよい（点滅や暗転程度）。

#### 演出の流れ（目安 1.0〜1.5 秒）

1. **一致パターンの発光** — 盤面上で課題と一致したマスが順に光る（グロー・白フラッシュ）
2. **課題パターンのハイライト** — 画面上部の課題表示が拡大しながら光る
3. **画面全体の光** — 背景が一瞬明るくなる（放射状グラデーション・ビネット解除）
4. **「クリア！」表示** — 光の輪（グロー）付きでスプリングイン
5. **操作ロック解除** — OK タップまたは短い待機後にメイン画面へ

#### 技術実装

| 用途 | 採用 |
|------|------|
| アニメーション基盤 | `react-native-reanimated` |
| 発光・グロー | Reanimated（opacity / scale）+ 半透明の白・黄系オーバーレイ |
| 触覚 | `expo-haptics`（クリア時に軽い成功フィードバック） |
| SE | `expo-av`（任意。未確定） |

紙吹雪・粒子系より **光・グロー・フラッシュ** を優先する。必要に応じて `@shopify/react-native-skia` でグロー強化を検討（初期は Reanimated のみで実装）。

#### UX ルール

- 演出中は盤面・行動ボタンを操作不可とする。
- 演出スキップ（タップでメインへ）は初期リリースでは任意（実装コストに応じて判断）。

## 7. ゲームオーバー

以下のいずれかを満たした時点でゲームオーバーとする（クリア判定を優先し、同一行動でクリアとゲームオーバーが同時に成立する場合は**クリア**とする）。

### 7.1 行動可能ターンの枯渇

- 残り行動可能ターンが **0** になり、かつ課題パターンが盤面上に存在しないときゲームオーバーとする。
- 行動実行後に残ターンが 0 になった場合、その行動の結果でクリアしていなければゲームオーバーとする。

### 7.2 盤面の満杯

- **操作可能マス**（お邪魔ブロック以外のマス）に空きがなくなった時点でゲームオーバーとする。
- 行動 B により新規ブロックを出現させられない状態も、満杯として扱う。

### 7.3 ゲームオーバー後の扱い

- プレイ中のステージ状態（オートセーブ）は削除する。
- メイン画面へ遷移する。再挑戦時はスタミナを消費する（チュートリアルステージを除く）。

## 8. ステージ設計方針

### 8.1 規模

- 初期リリース: **約100ステージ**

### 8.2 難易度カーブ（方針）


| 段階      | ステージ目安 | 内容                   |
| ------- | ------ | -------------------- |
| チュートリアル | 1〜5    | 極端に易しい。基本操作の導入       |
| 初級      | 6〜25   | 2〜3マスパターン、原色中心       |
| 中級      | 26〜60  | 二次色の混合必須、L字など3マス以上   |
| 上級      | 61〜90  | お邪魔ブロック、大きめパターン、盤面管理 |
| 最上級     | 91〜100 | 複合パターン、手数・NEXT読みの厳しさ |


### 8.3 ステージ1（確定仕様）


| 項目       | 値                            |
| -------- | ---------------------------- |
| 課題       | `[青][赤]`（左青・右赤）              |
| 初期配置     | 青ブロック 1 つ                    |
| NEXT（初期） | 赤（以降はランダム抽選）                 |
| 出現位置     | 固定（1〜2手でクリア可能な配置）            |
| 行動可能ターン  | 十分余裕のある値（例: 10）              |
| スタミナ     | 消費しない（チュートリアル）               |
| 目的       | 行動 B・NEXT 表示・パターンクリアの導入。混合不要 |


### 8.4 導入する要素の順序（目安）

1. 行動 B、NEXT、パターンクリア（ステージ1）
2. 出現位置ランダム（ステージ2〜）
3. 混合不要の重要性の再確認（隣接配置のみでクリアするステージ）
4. 行動 A（二次色を含む課題）（中〜初級）
5. 行動 C（スワップ）（中級）
6. お邪魔ブロック（中〜上級）
7. 黒・複雑混合色（上級）

## 9. ステージデータの保持方式

### 9.1 推奨: アプリ同梱（ローカル JSON）

初期リリースの約100ステージは、**アプリデータに同梱する JSON** として保持することを推奨する。


| 方式            | メリット                         | デメリット                    |
| ------------- | ---------------------------- | ------------------------ |
| **アプリ同梱（推奨）** | オフライン動作、読み込みが速い、サーバー不要、実装が単純 | ステージ追加のたびにアプリ更新が必要       |
| サーバー取得        | サーバー側でステージ追加・修正可能            | インフラ・認証・キャッシュ・オフライン対応が必要 |
| ハイブリッド        | 同梱分でオフライン、追加ステージを CDN 配信     | 実装・運用がやや複雑               |


**パズルゲーム100ステージ規模では、アプリ同梱が一般的。** データサイズは1ステージ数百バイト〜数 KB 程度の想定で、100ステージでも数 MB に満たない。

### 9.2 サーバー取得を検討するタイミング

以下の要件が出た場合に、サーバーまたは CDN 配信への移行を検討する。

- アプリ更新なしでステージを差し替え・追加したい
- イベント用ステージを頻繁に配信したい
- ユーザー生成ステージ（共有）を配信したい
- ランキング・不正対策でサーバー側検証が必要

初期リリース時点では不要。

### 9.3 ステージデータのファイル構成（案）

```
data/
  color_recipes.json    # 混合レシピ一覧
  stages/
    index.json          # ステージ一覧（id, ファイル名, 難易度メタデータ）
    stage_001.json
    stage_002.json
    ...
    stage_100.json
```

または単一ファイル `stages.json` に全ステージを配列で格納（100件程度なら十分現実的）。

### 9.4 ステージ JSON スキーマ（案）

```json
{
  "id": 1,
  "name": "はじめの一歩",
  "board": {
    "width": 4,
    "height": 3
  },
  "obstacles": [
    { "x": 0, "y": 0 }
  ],
  "initialBlocks": [
    { "x": 1, "y": 1, "color": "blue" }
  ],
  "pattern": {
    "cells": [
      { "dx": 0, "dy": 0, "color": "blue" },
      { "dx": 1, "dy": 0, "color": "red" }
    ]
  },
  "maxTurns": 10,
  "next": {
    "initial": "red"
  },
  "spawn": {
    "mode": "fixed",
    "fixedPosition": { "x": 2, "y": 1 }
  },
  "metadata": {
    "difficulty": 1,
    "par": 1,
    "isTutorial": true
  }
}
```


| フィールド                 | 説明                                   |
| --------------------- | ------------------------------------ |
| `board`               | 盤面サイズ                                |
| `obstacles`           | お邪魔ブロック座標                            |
| `initialBlocks`       | 初期配置                                 |
| `pattern`             | 課題パターン（相対座標）                         |
| `maxTurns`            | 行動可能ターン数（ステージ開始時に付与）                 |
| `next.initial`        | ステージ開始時の NEXT（以降はランダム抽選）             |
| `spawn.mode`          | `"fixed"` / `"random"` — 反対辺への出現位置   |
| `spawn.fixedPosition` | `fixed` 時の出現座標（ステージ1用）               |
| `metadata.par`        | 想定クリア手数（`maxTurns` 設計の参考値）           |
| `metadata.isTutorial` | `true` のときチュートリアル（スタミナ消費なし）。ステージ 1〜5 |


### 9.5 NEXT 抽選（確定）

- 行動 B 実行後の NEXT は、第4.5章のとおり**ランダム抽選**とする。
- ステージデータにシーケンス定義は持たない。
- チュートリアル（ステージ1〜5）でも NEXT 抽選はランダム。ステージ1のみ出現**位置**を固定する。

## 10. プレイテスト・バランス調整

実装後、以下の指標でバランスを調整する。


| 指標              | 目安                               |
| --------------- | -------------------------------- |
| クリア手数           | ステージ1: 1〜2 / 序盤: 3〜8 / 中盤: 10〜20 |
| ゲームオーバー率        | 序盤はほぼ0%、上級で初めて意味を持つ              |
| `maxTurns` の妥当性 | `par` の 1.5〜3 倍程度を目安に調整          |
| 行動 B の使用頻度      | 過多の場合は盤面圧が強すぎる可能性                |
| 行動 C の必要性       | スワップ不要で通せるステージが多すぎないか            |
| 誤混合             | 課題直前での誤った行動 A の発生頻度              |
| 黒の発生            | 意図せず黒が増えすぎないか                    |

### 13.3 アーキテクチャ方針

ゲームロジックと UI を分離し、Android 追加時もロジックを流用する。

```
app/                    # Expo Router 画面（メイン画面・ゲーム画面）
src/
  game/                 # 混合・スライド・スワップ・パターン判定（UI 非依存）
  storage/              # オートセーブ・スタミナ・進捗の永続化（抽象化し MMKV と差し替え可能に）
  ads/                  # 広告表示制御（開発中はモック実装）
  types/                # Stage, Recipe, SaveData 等の型定義
assets/
  data/
    color_recipes.json
    stages/             # ステージ JSON（アプリ同梱）
```

- `src/game/` は React Native に依存させない（ユニットテスト可能に保つ）。
- ステージデータは第9章のとおりアプリ同梱 JSON とする（初期リリース時点でサーバー不要）。

### 13.4 ビルド・リリースフロー（想定）

```
[Windows] 日常開発（Expo Go → Development Build）
    ↓
[EAS Build] クラウドで iOS .ipa / Android .apk|.aab 生成
    ↓
[TestFlight / 内部テスト] → App Store / Google Play 公開
```

開発フェーズの詳細は第18章を参照。

## 14. 画面構成

初期リリースでは以下の **2 画面** を用意する。

### 14.1 メイン画面（ステージ選択）


| 要素        | 説明                           |
| --------- | ---------------------------- |
| ステージ一覧    | 解放済みステージの選択。未解放ステージは非活性表示    |
| スタミナ表示    | 現在値 / 上限、次の自然回復までの残り時間       |
| リワード広告ボタン | スタミナ回復用（第16章）。スタミナが上限未満のとき表示 |
| バナー広告     | 画面下部等に常時表示（第16章）             |
| 続きから      | 中断中のステージがある場合、そのステージへ遷移する導線  |


集中力を要しない画面のため、**バナー広告を表示する**。

### 14.2 ゲーム画面


| 要素      | 説明                         |
| ------- | -------------------------- |
| 盤面      | 色ブロック・お邪魔ブロックの表示           |
| 課題パターン  | クリア条件の配色・配置を表示             |
| NEXT 表示 | 次に出現する色                    |
| 残りターン表示 | 残り行動可能ターン数                 |
| 行動ボタン   | A（混合）/ B（スライド＋方向）/ C（スワップ） |
| 戻る      | メイン画面へ（中断時はオートセーブ）         |


プレイ中は集中が必要なため、**バナー広告は非表示**とする。

### 14.3 画面遷移

```
メイン画面 ──（ステージ選択・スタミナ消費※）──→ ゲーム画面
※チュートリアル（ステージ1〜5）はスタミナ消費なし
ゲーム画面 ──（クリア / ゲームオーバー / 戻る）──→ メイン画面
```

- クリア時: 光多めのクリア演出（第6.4章）を再生後、ステージをクリア済みに記録し、次ステージを解放。メイン画面へ遷移。
- ゲームオーバー時: メイン画面へ遷移。再挑戦時は再度スタミナを消費する（チュートリアルを除く）。

## 15. スタミナ（挑戦権）

### 15.1 基本ルール


| 項目      | 値                        |
| ------- | ------------------------ |
| 上限      | **3**                    |
| 自然回復    | **10 分** ごとに **1** 回復    |
| 回復の上限   | 自然回復によりスタミナ上限（3）を超えない    |
| 消費タイミング | ステージ開始時に **1** 消費        |
| 消費の扱い   | クリア・ゲームオーバー・中断いずれでも返却しない |
| チュートリアル | **ステージ 1〜5** はスタミナを消費しない |


### 15.2 スタミナ不足時

- スタミナが 0 のとき、新規ステージ開始は不可とする。
- メイン画面に自然回復の残り時間と、リワード広告による回復手段を表示する。
- 中断中のステージへの「続きから」は、スタミナを消費しない（すでに開始済みのプレイを再開するため）。

### 15.3 時間回復の実装方針

- 最終回復処理時刻（または次回回復予定時刻）をローカルに保存する。
- アプリ起動時・メイン画面表示時・フォアグラウンド復帰時に経過時間を計算し、回復を反映する。
- スタミナが上限に達している間は回復タイマーを進めない。

### 15.4 リワード広告による回復

- リワード広告視聴完了時、スタミナを **1** 回復する。
- スタミナが上限（3）のときは、リワード広告ボタンを非活性または非表示とする。
- **1 日の視聴上限: 5 回**（確定）。

上限到達時は、ボタンを非活性にし「本日の回復上限に達しました」等を表示する。日付のリセットは **端末のローカル日付（0:00）** 基準とする。

## 16. 収益化（広告）

### 16.1 方針

収益化は **広告のみ** とする（初期リリース時点でアプリ内課金は行わない）。

### 16.2 広告の種類


| 種類         | 用途        | 表示場所                         |
| ---------- | --------- | ---------------------------- |
| **バナー広告**  | 常時掲載による収益 | メイン画面（ステージ選択）など、集中力を要しない画面   |
| **リワード広告** | スタミナ 1 回復 | メイン画面。ユーザーが明示的にボタンを押したときのみ表示 |


### 16.3 表示ルール

- **ゲーム画面ではバナー広告を表示しない。**
- リワード広告はユーザー操作によるオプトインのみ（自動再生しない）。
- リワード広告の視聴が完了した場合のみスタミナを付与する（途中離脱では付与しない）。
- 広告の読み込み失敗時は、スタミナを付与せずエラーメッセージを表示する。

### 16.4 広告 SDK（確定）

- 広告 SDK は **Google AdMob** を採用する。
- Expo 向けは `react-native-google-mobile-ads` を利用し、EAS Build でネイティブビルドする。
- 実装は第18章フェーズ4（最終段階）で行う。それまではモック広告で開発する。
- iOS の App Tracking Transparency（ATT）対応、プライバシーポリシー・広告表示に関する記載を App Store 提出時に用意する。

### 16.5 将来拡張（初期リリース対象外）

- アプリ内課金（スタミナ回復パック、広告非表示など）は初期リリースでは実装しない。必要になった段階で検討する。
- 買い切り状態で、スタミナ無限にする(広告はバナーのみになる)

## 17. 進捗のオートセーブ

### 17.1 方針

プレイ進捗はローカルに自動保存する。ユーザーによる手動セーブは不要とする。

永続化の実装は `src/storage/` で抽象化し、開発初期は AsyncStorage、本番前に MMKV へ差し替える（第18章参照）。

### 17.2 保存タイミング


| タイミング            | 保存内容                          |
| ---------------- | ----------------------------- |
| 行動 A / B / C 実行後 | プレイ中の盤面状態（ゲーム画面）              |
| ステージクリア時         | クリア済みステージ・解放ステージの更新。プレイ中状態を削除 |
| ゲームオーバー時         | プレイ中状態を削除                     |
| ゲーム画面を中断して戻る時    | プレイ中状態を保持                     |
| スタミナ変動時          | スタミナ値・回復タイマー・リワード視聴回数         |
| アプリ終了時           | 上記の最新状態を永続化                   |


### 17.3 保存データ（案）

```typescript
type SaveData = {
  version: number;
  unlockedStageId: number;
  clearedStages: number[];
  stageProgress: {
    stageId: number;
    board: BoardState;
    nextColor: string;
    lastAction: 'A' | 'B' | 'C' | null;
    remainingTurns: number;
  } | null;
  stamina: {
    current: number;
    lastRecoveryAt: number;       // Unix timestamp (ms)
  };
  rewardedAd: {
    dailyCount: number;
    lastResetDate: string;        // "YYYY-MM-DD" ローカル日付
  };
};
```

### 17.4 再開フロー

- アプリ起動時に `stageProgress` が存在すれば、メイン画面に「続きから」を表示する。
- 続きからはスタミナを消費せず、ゲーム画面へ直接遷移する。

## 18. 開発の進め方

### 18.1 方針

- 日常開発は **Expo Go** を用い、Windows 上で素早く反復する。
- **AdMob** と **MMKV** は最終フェーズまで導入を遅延させる（Expo Go 非対応のネイティブモジュールのため）。
- ゲームロジック（`src/game/`）と永続化・広告（`src/storage/`、`src/ads/`）はインターフェースで分離し、後から差し替え可能に保つ。

### 18.2 フェーズ一覧


| フェーズ           | 内容                                               | テスト手段             |
| -------------- | ------------------------------------------------ | ----------------- |
| **1. 基盤**      | プロジェクト初期化、Expo Router、型定義、ステージ JSON 読み込み         | Expo Go           |
| **2. コアゲーム**   | 混合・スライド・スワップ・パターン判定・ゲームオーバー・クリア                  | Expo Go + Jest    |
| **3. アプリ機能**   | メイン画面・ゲーム画面、スタミナ、オートセーブ（AsyncStorage）、ステージ100件   | Expo Go           |
| **4. ネイティブ統合** | MMKV 導入、AdMob 導入、広告モックの差し替え、Development Build 作成 | Development Build |
| **5. リリース準備**  | EAS Build（本番）、TestFlight、バランス調整、App Store 提出     | EAS Build         |


### 18.3 フェーズ1: 基盤

- `npx create-expo-app` でプロジェクト作成（TypeScript、Expo Router）。
- ディレクトリ構成は第13.3章に従う。
- `assets/data/` に `color_recipes.json`、ステージ JSON の雛形を配置。
- `src/game/` に盤面・色の型定義を置き、React Native へ依存しない。

### 18.4 フェーズ2: コアゲーム

- 行動 A / B / C のロジック、連続同一行動禁止、NEXT ランダム抽選を実装。
- パターン一致判定、ゲームオーバー（満杯・ターン枯渇）を実装。
- **Jest でユニットテスト**を書き、ロジックを Expo Go なしでも検証できるようにする。
- この段階では UI は最小限（デバッグ用盤面表示）でもよい。

### 18.5 フェーズ3: アプリ機能

- メイン画面（ステージ選択）・ゲーム画面を実装。
- スタミナ（上限3・10分回復・チュートリアル無料）を実装。
- オートセーブは **AsyncStorage** で実装（`src/storage/` に永続化インターフェースを定義）。
- 広告・リワードは **モック実装** とする。
  - バナー: プレースホルダー View を表示（「広告枠」）。
  - リワード: ボタン押下で即スタミナ +1（視聴上限5回/日のロジックはここで実装）。
- ステージ 1〜5（チュートリアル）および序盤ステージを Playtest し、バランス調整を開始。

**テスト方法（Expo Go）:**

```
expo start
→ Android エミュレータ / 実機、または iPhone の Expo Go アプリで QR スキャン
```

iOS 実機の動作確認は Mac なしで可能（Expo Go アプリ + 同一 Wi‑Fi）。

### 18.6 フェーズ4: ネイティブ統合（最終段階）

AdMob と MMKV は **ゲームプレイ・UI・バランスが固まってから** 導入する。

1. `expo-dev-client` を追加し、**Development Build** を EAS Build で作成。
2. `react-native-mmkv` を導入し、`src/storage/` の実装を AsyncStorage から差し替え。
3. `react-native-google-mobile-ads` を導入し、`src/ads/` のモックを AdMob 実装に差し替え。
4. AdMob のテスト広告 ID でバナー・リワードの表示・付与フローを検証。
5. iOS ATT（App Tracking Transparency）対応、プライバシーポリシー文案を用意。

**テスト方法（Development Build）:**

```
eas build --profile development --platform ios   # または android
expo start --dev-client
```

以降の日常開発は Dev Client 上でホットリロードする。Android は Windows 上で、iOS は EAS Build 成果物を実機にインストール。

### 18.7 フェーズ5: リリース準備

- EAS Build（本番プロファイル）で iOS `.ipa` を生成。
- TestFlight で実機テスト・バランス最終調整。
- App Store Connect へ提出（EAS Submit）。
- Android 版は同一コードベースから EAS Build で `.aab` を生成。

### 18.8 開発時の差し替えパターン

永続化・広告は実行環境に応じて実装を切り替える。

```typescript
// 例: src/storage/index.ts
export const storage = isDevClientOrProduction()
  ? mmkvStorage
  : asyncStorage;

// 例: src/ads/index.ts
export const ads = isDevClientOrProduction()
  ? adMobAds
  : mockAds;
```

`expo-constants` の `executionEnvironment` 等で Expo Go / Dev Client を判別する。

### 18.9 Expo Go でテストできないもの（フェーズ4まで延期）


| 機能         | 開発初期の代替       |
| ---------- | ------------- |
| AdMob バナー  | プレースホルダー View |
| AdMob リワード | モックボタンでスタミナ回復 |
| MMKV       | AsyncStorage  |
