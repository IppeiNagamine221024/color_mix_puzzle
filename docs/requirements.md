# Color Order — 要件定義

> **プロトタイプ実装済み（2026年6月時点）**  
> 本文書は初期構想に加え、Expo Go 上で動作する現行プロトタイプの仕様を反映している。  
> 特に **UI（ダークテーマ・スワイプスライド）**、**色管理（CMY 比・スタック添え字）**、**スタミナ上限 5** が初期案から変更されている。

## 1. 概要

**ゲーム名**: Color Order

三原色（シアン・マゼンタ・イエロー）をはじめとする色ブロックを盤面上で操作し、混合して、ステージごとの課題パターン（配色の配置）を盤面のどこかに再現したらクリアするパズルゲーム。

1ターンにつき1つの行動（A / B / C のいずれか）を実行する。**同一行動の連続禁止は設けない**（プロトタイプ確定）。

初期リリースでは約100ステージを用意する。

**プラットフォーム**: スマートフォン向けアプリ（iOS 先行リリース、Android を後続対応）。  
**収益化**: 広告（バナー・リワード）。  
**挑戦権**: スタミナ制（上限 **5**、10分回復 + リワード広告回復）。

---

## 2. 用語


| 用語          | 説明                                        |
| ----------- | ----------------------------------------- |
| ターン         | プレイヤーが行動（A / B / C のいずれか）を1回実行すること        |
| ブロック        | 盤面上の色付きマス。生の CMY 比とスタック添え字を持つ（第5章）         |
| スタック（添え字）   | ブロック左上の数字。比の GCD。同色の重なり量を表す（例: シアン `2`）    |
| お邪魔ブロック     | 移動・混合・スワップの対象にならない固定マス                    |
| 課題（パターン）    | クリア条件となる色と相対配置のテンプレート（正規化後の色 id で一致判定）   |
| NEXT        | 行動 B 実行時に出現する次のブロックの色。ゲーム画面に常時表示する        |
| 行動可能ターン     | ステージごとに与えられる行動回数の上限。行動 A / B / C ごとに1減少する |
| スタミナ        | ステージへ挑戦するために消費する挑戦権。上限・回復ルールは第15章参照       |
| チュートリアルステージ | ステージ 1〜5。スタミナを消費しない（第15章参照）               |


---

## 3. 盤面

- 盤面サイズ（幅・高さ）はステージごとに設定可能とする。
- マスには以下のいずれかが存在する。
  - 空き
  - 色ブロック（操作可能）
  - お邪魔ブロック（操作不可）
- 行動 B におけるブロックの移動は、2048 と同様に指定方向へ全ブロックがスライドし、壁・お邪魔ブロック・盤端で停止する。

---

## 4. 行動

各ターンで A / B / C のいずれか1つを実行する。直前と同じ行動でもよい。

### 4.1 行動 A — 混合

- 隣接（上下左右）する2つの色ブロックを選択し、1マスに混合する。
- 同色同士も混合可能。比は**正規化せず**成分ごとに足し算する（例: シアン `(1,0,0)` + シアン `(1,0,0)` → `(2,0,0)`）。盤面上の余剰ブロック整理に使える。
- 混合は、足し算後の比を正規化した色が `colors.json` に存在する場合のみ実行可能とする。該当色がなければ **MIX 不可**（操作として拒否する）。
- 混合後、もう一方のマスは空きになる。
- **黒の特殊ルール**
  - 黒と黒を混合した場合のみ、2マスとも消滅する（盤面から除去）。
  - 黒と黒以外の色の混合は不可。

### 4.2 行動 B — スライド＋出現

- 盤面全体を上下左右のいずれか1方向にスライドする。
- **操作**: 盤面エリア上の**スワイプ**で方向を指定し、即スライド実行する（方向パッド・B ボタンは使わない）。
- スライド後、移動方向の**反対側の辺**上の空きマスに、NEXT の色のブロックが1つ出現する（スタック `1` の原色）。
- 出現位置は、反対側の辺上の空きマスから**ランダム**に選ぶ。
- スライド実行後、NEXT を次の色に更新する（抽選ルールは第4.6章参照）。
- ゲーム画面上には常に現在の NEXT と残り行動可能ターン数を表示する。

#### ステージ1（チュートリアル）の例外

- NEXT の抽選は全ステージ共通で**3-bag 方式**（第4.6章）とするが、ステージ開始時の NEXT 初期値はステージデータで指定できる（ステージ1は `magenta`）。

### 4.3 行動 C — スワップ

- 隣接する2つの**操作可能な色ブロック**の位置を入れ替える。
- **操作**: 画面下部の「入替」ボタンでモードに入り、2タップで即実行する。
- 空きマス・お邪魔ブロックとのスワップは不可。

### 4.4 操作 UI（プロトタイプ確定）

| 行動 | UI |
| --- | --- |
| A 混合 | 下部「混合」ボタン → 盤面で2タップ（隣接・混合可のみ） |
| B スライド | 盤面エリアのスワイプ（上下左右）。混合・入替モード中は無効 |
| C 入替 | 下部「入替」ボタン → 盤面で2タップ（隣接ブロックのみ） |

- 混合・入替モード中はヒント文言を表示する（例: 「隣接する2つのブロックをタップ」）。
- スライド可能時は盤面に「↕↔ このエリアをスワイプでスライド」と表示する。
- 行動の拒否時は `Alert` で理由を表示する。

### 4.5 NEXT の更新タイミング

- NEXT の消費・更新は行動 B のときのみ行う。
- 行動 A / C の実行では NEXT は変化しない。

### 4.6 NEXT の抽選ルール（確定）

- 行動 B 実行後の NEXT 色は、**3-bag 方式**（テトリスの 7-bag と同様）で決定する（全ステージ共通）。
- 抽選対象は**原色**（シアン・マゼンタ・イエロー）3 色。3 色を 1 セットとしてシャッフルし、**セット内では各色が必ず 1 回ずつ**出現する。セットを使い切ったら再シャッフルする。
- 完全ランダム（独立抽選）ではなく、同じ色が長く続く偏りを抑えることを目的とする。
- ステージ開始時の NEXT 初期値のみ、ステージデータの `next.initial` で指定できる（バッグとは独立）。
- 抽選対象色の拡張（二次色を含める等）はステージデータまたはゲーム設定で上書き可能とする（初期リリースでは原色のみ）。

### 4.7 行動可能ターン

- 各ステージは `maxTurns`（行動可能ターン数）を持つ。
- 行動 A / B / C を1回実行するたびに、残り行動可能ターンを **1** 減少させる。
- 行動の実行が拒否された場合（混合不可など）は減少しない。

## 5. 色と混合ルール

### 5.1 基本方針

- 混合は一般的な絵の具（減法混色）の考え方に基づく。
- **三原色**は **シアン（C）・マゼンタ（M）・イエロー（Y）** とする（RGB の赤・青・黄ではない）。
- 各色は CMY 比 `(C, M, Y)` を整数比で持つ。原色は `(1,0,0)` / `(0,1,0)` / `(0,0,1)`。
- ゲーム内で使用する色は、すべて `colors.json` に **id・表示名・ratio・hex** として定義する。
- 原色・二次色・三次色など、カタログに定義され、原色から混合で到達可能な色のみ盤面・課題・NEXT に登場しうる。

### 5.2 ブロックの表現

- 盤上の各ブロックは **生の CMY 比** `(C,M,Y)` を保持する（混合のたびに正規化しない）。
- **見た目の色名**は、比を正規化して `colors.json` と照合した id で決まる。
- **添え字（スタック数）**は比の最大公約数とし、ブロック左上に表示する（例: `(2,0,0)` → シアンに添え字 `2`）。
- 盤上ブロックの各成分は **0〜3** まで。いずれかが 4 以上になる混合は不可。
- 課題パターンの一致判定は**正規化後の色 id** で行う（添え字は問わない）。

### 5.3 混合の計算

1. 混合する 2 ブロックの CMY 比を、成分ごとに足し算する（**正規化しない**）。
2. いずれかの成分が 4 以上なら混合不可。
3. 合計比を正規化し、カタログに該当色があれば混合可能。結果ブロックは**正規化前の合計比**を保持する。
4. 正規化後に該当色がなければ混合不可。

**例**

| 混合 | 計算（生の比） | 表示 |
| --- | --- | --- |
| シアン(1) + シアン(1) | `(1,0,0)+(1,0,0)=(2,0,0)` | シアン(2) |
| シアン(2) + イエロー(1) | `(2,0,0)+(0,0,1)=(2,0,1)` | ティール(1) |
| マゼンタ + イエロー | `(0,1,0)+(0,0,1)=(0,1,1)` | 赤(1) |
| 赤 + シアン | `(0,1,1)+(1,0,0)=(1,1,1)` | 黒(1) |

### 5.4 混合の可否


| 条件 | 結果 |
| --- | --- |
| 合計比の正規化結果がカタログに存在し、各成分 ≤ 3 | 混合可能 |
| 正規化後の比がカタログに存在しない | 混合不可 |
| 合計比のいずれかの成分が 4 以上 | 混合不可 |
| 同色同士（黒以外） | 混合可能（スタックが増える場合あり） |
| 黒 + 黒 | 両方消滅（行動 A） |
| 黒 + 黒以外 | 混合不可 |


### 5.5 黒

- 黒は CMY 比 `(1,1,1)` として定義する（三原色が等量比で揃った状態）。
- 例: 赤 `(0,1,1)` + シアン `(1,0,0)` → 黒。緑 + マゼンタ、青 + イエローなども同様。
- 黒は不要色の安全弁ではなく、主に盤面圧力・ペナルティとして機能する。黒同士の除去は意図的な盤面管理手段とする。

### 5.6 色データ（`colors.json`）

各色は次の形式で定義する。

```json
{ "id": "red", "label": "赤", "ratio": [0, 1, 1], "hex": "#F44336" }
```

| フィールド | 説明 |
| --- | --- |
| `id` | ステージ JSON 等で参照する識別子 |
| `label` | 画面表示用の日本語名 |
| `ratio` | CMY 比 `[C, M, Y]`（正規化済みの整数比） |
| `hex` | ブロック表示色（UI 用。ロジックとは独立） |

**初期リリース時点の色一覧**

各色の ratio は **正規化済み** とし、**いずれかの成分も 3 を超えない**（最大値 3）。

**運用ルール**

- 色を追加するときは `colors.json` に 1 行追加する。組み合わせごとのレシピ定義は不要。
- 正規化後の ratio が重複する色は登録できない（ロード時に検証）。
- ratio の各成分は **0〜3** の整数とし、**最大公約数 1**（最簡比）で保存する。非正規化または成分が 3 を超える色はロード時にエラーとする。
- 原色（NEXT 抽選対象）は ratio から自動判定する（成分の和が 1 の色）。
- 混合ロジックは `src/game/colorRatio.ts`（比の演算）と `src/game/colorCatalog.ts`（カタログ読み込み・混合判定）に実装する。

**廃止**: 旧方式の `color_recipes.json`（組み合わせごとのレシピ列挙）は使用しない。

## 6. クリア条件

- ステージごとに定義された**課題パターン**が、盤面上の**どこか**に出現した瞬間に即クリアとする。
- パターン一致時に追加の確認操作（提出ボタンなど）は不要。
- クリア成立後は **光を中心とした派手めの演出**（第6.4章）を再生し、オーバーレイ経由でメイン画面へ遷移する。

### 6.1 パターン一致ルール（デフォルト）


| 項目        | ルール                 |
| --------- | ------------------- |
| 回転        | 不可                  |
| 反転        | 不可                  |
| 配置        | 盤面の任意位置で一致すればよい     |
| 課題外のマス    | パターンに含まれないマスは何色でもよい |
| スタック添え字   | パターン一致には影響しない（色 id のみ比較） |
| お邪魔ブロック   | パターンに含めない           |
| 最小パターンサイズ | 2マス                 |


### 6.2 パターンの向き

- 課題の表示どおりの相対配置で一致する必要がある。
- 例: `[シアン][マゼンタ]` は**左がシアン・右がマゼンタ**の横並び。
- 縦並びの `[シアン]` / `[マゼンタ]` は別課題として定義する（横並びとは別物）。

### 6.3 課題パターンのデータ表現

- 課題は「色付きセルの相対座標」のリストとして保持する。
- 例（横2マス `[シアン][マゼンタ]`）:

```json
{
  "cells": [
    { "dx": 0, "dy": 0, "color": "cyan" },
    { "dx": 1, "dy": 0, "color": "magenta" }
  ]
}
```

### 6.4 クリア演出（確定）

クリア演出は **光多め・派手目** を方針とする。ゲームオーバー演出は簡素でよい（点滅や暗転程度）。

#### 演出の流れ（目安 1.0〜1.5 秒）

1. **一致パターンの発光** — 盤面上で課題と一致したマスが順に光る（グロー・白フラッシュ）
2. **課題パターンのハイライト** — 画面上部の課題表示が拡大しながら光る
3. **画面全体の光** — 背景が一瞬明るくなる（放射状グラデーション・ビネット解除）
4. **「クリア！」表示** — 光の輪（グロー）付きでスプリングイン
5. **操作ロック解除** — OK タップまたは短い待機後にメイン画面へ

#### 技術実装（プロトタイプ）

| 用途 | 採用 |
|------|------|
| 盤面アニメーション（スライド・混合・入替） | React Native 標準 `Animated` API（Expo Go 互換） |
| クリアオーバーレイ | `react-native-reanimated`（グロー・パルス） |
| スタミナ消費演出 | React Native 標準 `Animated` API |
| 発光・グロー | 半透明オーバーレイ + opacity / scale |
| 触覚 | `expo-haptics`（任意。未実装） |
| SE | `expo-av`（任意。未確定） |

紙吹雪・粒子系より **光・グロー・フラッシュ** を優先する。

#### UX ルール

- 演出中は盤面・操作をロックする（アニメーション完了まで）。
- 演出スキップ（タップでメインへ）は初期リリースでは任意（実装コストに応じて判断）。

## 7. ゲームオーバー

以下のいずれかを満たした時点でゲームオーバーとする（クリア判定を優先し、同一行動でクリアとゲームオーバーが同時に成立する場合は**クリア**とする）。

### 7.1 行動可能ターンの枯渇

- 残り行動可能ターンが **0** になり、かつ課題パターンが盤面上に存在しないときゲームオーバーとする。
- 行動実行後に残ターンが 0 になった場合、その行動の結果でクリアしていなければゲームオーバーとする。

### 7.2 盤面の満杯

- **操作可能マス**（お邪魔ブロック以外のマス）に空きがなくなった時点でゲームオーバーとする。
- 行動 B により新規ブロックを出現させられない状態も、満杯として扱う。

### 7.3 ゲームオーバー後の扱い

- プレイ中のステージ状態（オートセーブ）は削除する。
- メイン画面へ遷移する。再挑戦時はスタミナを消費する（チュートリアルステージを除く）。

## 9. ステージデータの保持方式

### 9.1 推奨: アプリ同梱（ローカル JSON）

初期リリースの約100ステージは、**アプリデータに同梱する JSON** として保持することを推奨する。


| 方式            | メリット                         | デメリット                    |
| ------------- | ---------------------------- | ------------------------ |
| **アプリ同梱（推奨）** | オフライン動作、読み込みが速い、サーバー不要、実装が単純 | ステージ追加のたびにアプリ更新が必要       |
| サーバー取得        | サーバー側でステージ追加・修正可能            | インフラ・認証・キャッシュ・オフライン対応が必要 |
| ハイブリッド        | 同梱分でオフライン、追加ステージを CDN 配信     | 実装・運用がやや複雑               |


**パズルゲーム100ステージ規模では、アプリ同梱が一般的。** データサイズは1ステージ数百バイト〜数 KB 程度の想定で、100ステージでも数 MB に満たない。

### 9.2 サーバー取得を検討するタイミング

以下の要件が出た場合に、サーバーまたは CDN 配信への移行を検討する。

- アプリ更新なしでステージを差し替え・追加したい
- イベント用ステージを頻繁に配信したい
- ユーザー生成ステージ（共有）を配信したい
- ランキング・不正対策でサーバー側検証が必要

初期リリース時点では不要。

### 9.3 ステージデータのファイル構成

```
assets/data/

  stages.json           # 全ステージ（単一 JSON、約100件）
```

`scripts/generate-stages.js` で `stages.json` を再生成できる。

### 9.4 ステージ JSON スキーマ（案）

```json
{
  "id": 1,
  "name": "はじめの一歩",
  "board": {
    "width": 4,
    "height": 3
  },
  "obstacles": [
    { "x": 0, "y": 0 }
  ],
  "initialBlocks": [
    { "x": 2, "y": 1, "color": "cyan" }
  ],
  "pattern": {
    "cells": [
      { "dx": 0, "dy": 0, "color": "cyan" },
      { "dx": 1, "dy": 0, "color": "magenta" }
    ]
  },
  "maxTurns": 10,
  "next": {
    "initial": "magenta"
  },
  "spawn": {
    "mode": "fixed",
    "fixedPosition": { "x": 2, "y": 1 }
  },
  "metadata": {
    "difficulty": 1,
    "par": 1,
    "isTutorial": true
  }
}
```


| フィールド                 | 説明                                   |
| --------------------- | ------------------------------------ |
| `board`               | 盤面サイズ                                |
| `obstacles`           | お邪魔ブロック座標                            |
| `initialBlocks`       | 初期配置                                 |
| `pattern`             | 課題パターン（相対座標）                         |
| `maxTurns`            | 行動可能ターン数（ステージ開始時に付与）                 |
| `next.initial`        | ステージ開始時の NEXT（以降は 3-bag 抽選）             |
| `spawn.mode`          | `"fixed"` / `"random"` — 反対辺への出現位置   |
| `spawn.fixedPosition` | `fixed` 時の出現座標（デバッグ・検証用）               |
| `metadata.par`        | 想定クリア手数（`maxTurns` 設計の参考値）           |
| `metadata.isTutorial` | `true` のときチュートリアル（スタミナ消費なし）。ステージ 1〜5 |


### 9.5 NEXT 抽選（確定）

- 行動 B 実行後の NEXT は、第4.5章のとおり**3-bag 方式**とする。
- ステージデータにシーケンス定義は持たない（バッグ状態はプレイ中セッション／オートセーブに保持）。
- チュートリアル（ステージ1〜5）でも NEXT 抽選は 3-bag。出現位置は全ステージランダム。

## 15. スタミナ（挑戦権）

### 15.1 基本ルール


| 項目      | 値                        |
| ------- | ------------------------ |
| 上限      | **5**                    |
| 自然回復    | **10 分** ごとに **1** 回復    |
| 回復の上限   | 自然回復によりスタミナ上限（5）を超えない    |
| 消費タイミング | 非チュートリアルステージ開始時に **1** 消費（**メイン画面**で演出付き） |
| 消費の扱い   | クリア・ゲームオーバー・中断いずれでも返却しない |
| チュートリアル | **ステージ 1〜5** はスタミナを消費しない |


### 15.2 スタミナ不足時

- スタミナが 0 のとき、新規ステージ開始は不可とする。
- メイン画面に自然回復の残り時間と、リワード広告による回復手段を表示する。
- 中断中のステージへの「続きから」は、スタミナを消費しない（すでに開始済みのプレイを再開するため）。

### 15.3 時間回復の実装方針

- 最終回復処理時刻（または次回回復予定時刻）をローカルに保存する。
- アプリ起動時・メイン画面表示時・フォアグラウンド復帰時に経過時間を計算し、回復を反映する。
- スタミナが上限に達している間は回復タイマーを進めない。

### 15.4 リワード広告による回復

- リワード広告視聴完了時、スタミナを **1** 回復する。
- スタミナが上限（5）のときは、リワード広告ボタンを非活性または非表示とする。
- **1 日の視聴上限: 5 回**（確定）。

上限到達時は、ボタンを非活性にし「本日の回復上限に達しました」等を表示する。日付のリセットは **端末のローカル日付（0:00）** 基準とする。

## 16. 収益化（広告）

### 16.1 方針

収益化は **広告のみ** とする（初期リリース時点でアプリ内課金は行わない）。

### 16.2 広告の種類


| 種類         | 用途        | 表示場所                         |
| ---------- | --------- | ---------------------------- |
| **バナー広告**  | 常時掲載による収益 | メイン画面（ステージ選択）など、集中力を要しない画面   |
| **リワード広告** | スタミナ 1 回復 | メイン画面。ユーザーが明示的にボタンを押したときのみ表示 |


### 16.3 表示ルール

- **ゲーム画面ではバナー広告を表示しない。**
- リワード広告はユーザー操作によるオプトインのみ（自動再生しない）。
- リワード広告の視聴が完了した場合のみスタミナを付与する（途中離脱では付与しない）。
- 広告の読み込み失敗時は、スタミナを付与せずエラーメッセージを表示する。

### 16.4 広告 SDK（確定）

- 広告 SDK は **Google AdMob** を採用する。
- Expo 向けは `react-native-google-mobile-ads` を利用し、EAS Build でネイティブビルドする。
- 実装は第18章フェーズ4（最終段階）で行う。それまではモック広告で開発する。
- iOS の App Tracking Transparency（ATT）対応、プライバシーポリシー・広告表示に関する記載を App Store 提出時に用意する。

### 16.5 将来拡張（初期リリース対象外）

- アプリ内課金（スタミナ回復パック、広告非表示など）は初期リリースでは実装しない。必要になった段階で検討する。
- 買い切り状態で、スタミナ無限にする(広告はバナーのみになる)

## 17. 進捗のオートセーブ

### 17.1 方針

プレイ進捗はローカルに自動保存する。ユーザーによる手動セーブは不要とする。

永続化の実装は `src/storage/` で抽象化し、開発初期は AsyncStorage、本番前に MMKV へ差し替える（第18章参照）。

### 17.2 保存タイミング


| タイミング            | 保存内容                          |
| ---------------- | ----------------------------- |
| 行動 A / B / C 実行後 | プレイ中の盤面状態（ゲーム画面）              |
| ステージクリア時         | クリア済みステージ・解放ステージの更新。プレイ中状態を削除 |
| ゲームオーバー時         | プレイ中状態を削除                     |
| ゲーム画面を中断して戻る時    | プレイ中状態を保持                     |
| スタミナ変動時          | スタミナ値・回復タイマー・リワード視聴回数         |
| アプリ終了時           | 上記の最新状態を永続化                   |


### 17.3 保存データ（案）

```typescript
type SaveData = {
  version: number;               // 現在: 2（ブロックは ratio 保持）
  unlockedStageId: number;
  clearedStages: number[];
  stageProgress: {
    stageId: number;
    board: Board;                // ブロックは { kind: 'block', ratio: [C,M,Y] }
    nextColor: string;
    nextBag: string[];
    lastAction: 'A' | 'B' | 'C' | null;
    remainingTurns: number;
  } | null;
  stamina: {
    current: number;
    lastRecoveryAt: number;
  };
  rewardedAd: {
    dailyCount: number;
    lastResetDate: string;
  };
};
```

### 17.4 再開フロー

- アプリ起動時に `stageProgress` が存在すれば、メイン画面に「続きから」を表示する。
- 続きからはスタミナを消費せず、ゲーム画面へ直接遷移する。

## 18. 開発の進め方

### 18.1 方針

- 日常開発は **Expo Go** を用い、Windows 上で素早く反復する。
- **AdMob** と **MMKV** は最終フェーズまで導入を遅延させる（Expo Go 非対応のネイティブモジュールのため）。
- ゲームロジック（`src/game/`）と永続化・広告（`src/storage/`、`src/ads/`）はインターフェースで分離し、後から差し替え可能に保つ。

### 18.2 フェーズ一覧


| フェーズ           | 内容                                               | テスト手段             |
| -------------- | ------------------------------------------------ | ----------------- |
| **1. 基盤**      | プロジェクト初期化、Expo Router、型定義、ステージ JSON 読み込み         | Expo Go           |
| **2. コアゲーム**   | 混合・スライド・スワップ・パターン判定・ゲームオーバー・クリア                  | Expo Go + Jest    |
| **3. アプリ機能**   | メイン画面・ゲーム画面、スタミナ、オートセーブ（AsyncStorage）、ステージ100件   | Expo Go           |
| **4. ネイティブ統合** | MMKV 導入、AdMob 導入、広告モックの差し替え、Development Build 作成 | Development Build |
| **5. リリース準備**  | EAS Build（本番）、TestFlight、バランス調整、App Store 提出     | EAS Build         |


### 18.3 フェーズ1: 基盤

- `npx create-expo-app` でプロジェクト作成（TypeScript、Expo Router）。
- ディレクトリ構成は第13.3章に従う。
- `assets/data/` に `colors.json`、ステージ JSON の雛形を配置。
- `src/game/` に盤面・色の型定義を置き、React Native へ依存しない。

### 18.4 フェーズ2: コアゲーム

- 行動 A / B / C のロジック、NEXT 3-bag 抽選を実装。
- パターン一致判定、ゲームオーバー（満杯・ターン枯渇）を実装。
- **Jest でユニットテスト**を書き、ロジックを Expo Go なしでも検証できるようにする。
- この段階では UI は最小限（デバッグ用盤面表示）でもよい。

### 18.5 フェーズ3: アプリ機能

- メイン画面（ステージ選択）・ゲーム画面を実装。
- スタミナ（上限 **5**・10分回復・チュートリアル無料・メイン画面での消費演出）を実装。
- オートセーブは **AsyncStorage** で実装（`src/storage/` に永続化インターフェースを定義）。
- 広告・リワードは **モック実装** とする。
  - バナー: プレースホルダー View を表示（「広告枠」）。
  - リワード: ボタン押下で即スタミナ +1（視聴上限5回/日のロジックはここで実装）。
- ステージ 1〜5（チュートリアル）および序盤ステージを Playtest し、バランス調整を開始。

**開発用ツール（リリース対象外）**

- `src/dev/config.ts` の `DEV_TOOLS_ENABLED` により、メイン画面にスタミナ即回復ボタンを表示できる。
- 本番ビルド前に無効化する。

**テスト方法（Expo Go）:**

```
expo start
→ Android エミュレータ / 実機、または iPhone の Expo Go アプリで QR スキャン
```

iOS 実機の動作確認は Mac なしで可能（Expo Go アプリ + 同一 Wi‑Fi）。

### 18.6 フェーズ4: ネイティブ統合（最終段階）

AdMob と MMKV は **ゲームプレイ・UI・バランスが固まってから** 導入する。

1. `expo-dev-client` を追加し、**Development Build** を EAS Build で作成。
2. `react-native-mmkv` を導入し、`src/storage/` の実装を AsyncStorage から差し替え。
3. `react-native-google-mobile-ads` を導入し、`src/ads/` のモックを AdMob 実装に差し替え。
4. AdMob のテスト広告 ID でバナー・リワードの表示・付与フローを検証。
5. iOS ATT（App Tracking Transparency）対応、プライバシーポリシー文案を用意。

**テスト方法（Development Build）:**

```
eas build --profile development --platform ios   # または android
expo start --dev-client
```

以降の日常開発は Dev Client 上でホットリロードする。Android は Windows 上で、iOS は EAS Build 成果物を実機にインストール。

### 18.7 フェーズ5: リリース準備

- EAS Build（本番プロファイル）で iOS `.ipa` を生成。
- TestFlight で実機テスト・バランス最終調整。
- App Store Connect へ提出（EAS Submit）。
- Android 版は同一コードベースから EAS Build で `.aab` を生成。

### 18.8 開発時の差し替えパターン

永続化・広告は実行環境に応じて実装を切り替える。

```typescript
// 例: src/storage/index.ts
export const storage = isDevClientOrProduction()
  ? mmkvStorage
  : asyncStorage;

// 例: src/ads/index.ts
export const ads = isDevClientOrProduction()
  ? adMobAds
  : mockAds;
```

`expo-constants` の `executionEnvironment` 等で Expo Go / Dev Client を判別する。
